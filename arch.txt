Architecture decision

Use source-specific fetchers first, generic fallback second.

That means:

if URL matches Greenhouse, use Greenhouse fetcher
if URL matches Lever, use Lever fetcher
else use generic HTML fetcher

This is the right boring design.

Why this matters

ATS scoring depends on:

real requirements
role wording
seniority
tech stack
constraints like visa/remote/location

You cannot do that reliably from a search snippet.

Module 5: LLM insights layer

Only runs for shortlisted or borderline jobs.

Your prior design already says: deterministic first, model only for shortlisted jobs. That is exactly right.

What it does

Input:

compact normalized job
compact candidate profile
ATS rule score output

Output:

strengths_to_emphasize
gaps_to_address
summary_angle
bullet_focus
interview_hooks optional
Key decision

This module must be bounded.

Do not send the entire resume and full JD blindly every time.
Instead build a compact prompt packet:

job summary
extracted requirements
candidate fact summary
component scores
missing gaps
target output JSON only

That keeps token usage low.
